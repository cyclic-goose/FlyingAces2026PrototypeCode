Nothing in software jumps out as a dragging cause. A dragging wheel is most likely hardware:

Encoder offset slightly off — if one module's CANcoder offset is wrong, the wheel could sit at a slight angle and drag. You could re-zero the offsets in Tuner X to verify.
Mechanical friction — a tight bearing, misaligned azimuth, or over-tightened bolt
Coupling ratio — your kCoupleRatio of 5.4 compensates for drive/steer coupling. If this doesn't match your actual module, it could cause slight drift, but usually not dragging.
To diagnose, I'd suggest:

Put the robot on blocks and check if all 4 modules spin freely by hand
In Tuner X, verify each module's CANcoder reads ~0 when wheels face forward
Check if the dragging wheel's steer motor draws noticeably more current than the others (visible in Phoenix Tuner self-test or signal logging)
The software side looks clean — I'd lean toward hardware (mechanical binding or an encoder offset that needs re-zeroing).



Limelight-to-RoboRIO Connection
Physical Setup
Ethernet cable from the Limelight to your robot's network switch (or directly to the radio's extra port)
Power the Limelight via the VRM (12V/500mA regulated output) or a dedicated power source
Configure the Limelight's IP
Connect to the robot's network (via WiFi or tethered)
Open a browser and go to http://limelight.local:5801 (the Limelight web UI)
Go to Settings > Networking and set:
IP: 10.TE.AM.11 (e.g., if your team number is 1234, use 10.12.34.11)
Subnet: 255.255.255.0
Gateway: 10.TE.AM.1
Team number must match your RoboRIO's team number
Verify Communication
The fastest way to confirm data is flowing:

NetworkTables check — Deploy your code, then open Shuffleboard or AdvantageScope. Look for the limelight table. You should see tv, tx, ty, and botpose_wpiblue entries updating.

Your code's logging — Since your Limelight.java logs to AdvantageKit every cycle, check AdvantageScope for:

Limelight/hasTarget — should flip to true when you point at an AprilTag
Limelight/tx / Limelight/ty — should change as you move the camera
Limelight/BotPose — should show a valid field position when tags are visible
Quick smoke test in the Limelight UI — http://limelight.local:5801 shows a live camera feed. If you see AprilTag detections drawn on the feed, the camera side is working. If NetworkTables data also appears in Shuffleboard, the connection to the RoboRIO is confirmed.

Common Gotchas
Wrong IP subnet — The Limelight, RoboRIO, and radio must all be on 10.TE.AM.x. If the Limelight is still on its factory default (10.0.0.x), it won't talk to the RoboRIO on the field network.
Team number mismatch — Set the team number in both the Limelight web UI and the RoboRIO imaging tool.
Firewall on your laptop — If you can't reach limelight.local:5801, try the static IP directly (e.g., http://10.12.34.11:5801).
Pipeline not set to AprilTags — In the Limelight UI, make sure the active pipeline is configured for AprilTag detection, not retroreflective or color.

Check NetworkTables — open Shuffleboard or AdvantageScope and confirm limelight/tx, limelight/tv, etc. show up with correct values
Tune PID — your current gains (kP=0.1, kD=0.005) are a reasonable starting point. If the robot oscillates around the target, lower kP. If it's too sluggish, raise it. Test on carpet, not on a slick surface

Goal Posts with Two Tags
Since goal posts have two AprilTags, the Limelight will lock onto whichever is closer/more visible. This is fine for alignment — both tags on the same post will give you roughly the same tx heading. The "Sort Mode: Closest" setting ensures consistent behavior as you approach.



Field map — Make sure you've selected the correct game's AprilTag field map (2025/2026) so the Limelight knows where each tag is physically located on the field. This is critical for botpose to return accurate positions.

Camera resolution — Higher resolution improves tag detection at distance but increases latency. A common choice is 1280x960 for a good balance.

Pipeline index — Make sure the pipeline you configured is pipeline 0 (or whichever your code expects — your code doesn't set a pipeline, so it defaults to 0).

Stream mode — Set to "Standard" unless you have a second camera.

IP address — Your code uses the default NetworkTable name "limelight". If you renamed your Limelight (e.g., "limelight-front"), you'd need to update line 39 to match.

3D mode — Make sure "3D" or "Solve PnP" is enabled in the pipeline, since your code uses botpose_orb_wpiblue which requires the 3D solver.
